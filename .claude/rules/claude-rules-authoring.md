# `.claude/` 内ルールの書き方

`.claude/CLAUDE.md` と `.claude/rules/*.md` を**書く・更新する**時に従う。

`.claude/` 配下は Claude Code が**自動でコンテキストに読み込む**：

- `CLAUDE.md`：毎セッション読み込まれる
- `rules/*.md`：frontmatter `paths:` にマッチするファイルを編集する時に読み込まれる

つまり**ここに書いたルールは Claude が必ず目にする**。逆にリンク先のファイルは読みに行かない可能性が高い。この前提でルールを書く。

---

## 1. ルール本体は直接記載する（リンクで逃さない）

ルールを「他ファイルに書いてある」とリンクで誘導しない。**Claude はリンクを必ず辿るとは限らない**ため、ルール本体は load されるそのファイル内で完結させる。

リンクを使ってよいのは以下に限定：

- **採用根拠 / 設計判断**：「採用根拠は [ADR 00XX] を参照」（ADR は読まなくても規約には従える）
- **実装 SSoT のコードファイル**：「機械強制 SSoT は `commitlint.config.ts`」（Claude は必要時に直接そのファイルを開ける）
- **より長い解説 / 具体例 / フローチャート**：「詳細は <要件定義書> を参照」（基本ルールはこのファイル内で読み切れる前提）

---

## 2. Claude が読み取りやすい構造にする

- 見出しで区切り、トピックを明示する
- 規約は**表 / 箇条書き**で列挙する。長文の散文は避ける
- 適用テンプレートはそのまま貼れる**コードブロック**で示す
- 1 ルールにつき動詞（「〜する」「〜しない」）が含まれる 1 文を基準とし、根拠は短く後置する

---

## 3. 直接転記の判断（SSoT が他にある場合）

要件定義書 / 機械強制設定ファイル（`commitlint.config.ts` / `.syncpackrc.ts` 等）に SSoT がある規約を `.claude/` 内に書く場合：

### 直接転記する基準

- 適用する：短い規約・表・列挙（コミット scope 表、ブランチ命名、設定ファイル形式の優先順位、レイヤ別実装ルール 等）
- 適用しない：長文の前提原則、採用根拠、代替案、フローチャート、トレードオフ議論 — これらは要件定義書 / ADR にだけ書く

### 転記時の必須事項

- 該当セクション冒頭に「**SSoT は <要件定義書 X.md / 設定ファイル>、本セクションは Claude が直接読む用の縮約版（SSoT 更新時はここも合わせて更新する）**」と明記
- SSoT を更新したら `.claude/` 内のミラーも同じ PR で更新する（差分が一致することを PR レビューで人間が確認）

---

## 4. `.claude/` 内ファイル間の役割分担

複数の rules が同じ paths にマッチして両方 load される場合は、役割分担で重複を避ける：

| ファイル | 役割 |
|---|---|
| `.claude/CLAUDE.md` | プロジェクト全体に効かせたい横断規約（コミット形式 / ブランチ命名 / Git 操作禁止 / 設定ファイル形式の優先順位 / コード品質ツール一覧 等） |
| `.claude/rules/claude-rules-authoring.md`（本ファイル） | `.claude/` 内ルールの書き方そのもの |
| `.claude/rules/docs-rules.md` | `docs/`（ADR / 要件定義書）と README / CONTRIBUTING.md の記述ルール（SSoT 原則 / ADR Decision の型分類 / 5 バケット構造 / Mermaid 規約 等） |
| `.claude/rules/{backend,frontend,worker,drizzle,prompts}.md` | レイヤ別実装ルール |

役割が重なるトピックが出てきたら、片方を「正」とし、他方からは「<役割名> は <ファイル名> を参照」と一文で誘導する（リンクではなくファイル名をテキストで明示し、Claude が直接そのファイルを開けるようにする）。
